# Code Audit — 2026-07-03

Full-repo audit of the autonomy stack (`ros2_ws/src/shark_isr_*`), SITL test harness (`sim/`),
launch scripts (`scripts/`), and the training pipeline (`training/`). Vendored `px4_msgs` was
excluded. Every finding below was either fixed and verified, or deliberately left and flagged.

## Fixed

### 1. `sim/tests/run_tests.sh` — test runner aborted after the first passing test (critical)

`((PASS++))` under `set -euo pipefail` is a classic bash trap: the arithmetic post-increment
evaluates to the *old* value, so with `PASS=0` the expression returns exit status 1 and `set -e`
kills the script. The runner therefore exited silently after the first test instead of running
the full T06–T11 sequence. Same defect on `((FAIL++))`.

**Fix:** replaced with `PASS=$((PASS + 1))` / `FAIL=$((FAIL + 1))`, with a comment explaining why.

### 2. `guidance_node.py` — detection with no vehicle state sent the orbit to the ENU origin

If a `Detection` arrived before any `VehicleState` had been received, both fallback branches in
`_cb_detection` were skipped and the orbit centre defaulted to `(0.0, 0.0)` — the aircraft would
enter TRACK and fly to the launch point instead of the target. Low likelihood in practice
(detections normally imply an active mission), but a genuine fly-to-origin path.

**Fix:** guard at the top of `_cb_detection`: no vehicle state → log a warning and drop the
detection.

### 3. `autopilot_bridge.py` — unknown setpoint type published a zeroed TrajectorySetpoint

The heartbeat's type dispatch (`TYPE_POSITION` / `TYPE_VELOCITY` / `TYPE_ORBIT`) had no else
branch. A `GuidanceSetpoint` with any other `setpoint_type` value fell through every branch and
published a default-constructed `TrajectorySetpoint` — position `[0, 0, 0]` in NED, i.e. fly to
origin at ground level.

**Fix:** unknown types are now rejected at ingest in `_cb_guidance_setpoint` with an error log.
The stored setpoint stays `None`/stale, so the heartbeat keeps flying the safe velocity-hold path.

### 4. `guidance_node.py` — dead code removed

`_publish_hold()` and its `_hold_target` latch state were never called from anywhere (the IDLE
phase intentionally publishes nothing and lets the bridge's stale-setpoint hold take over).
Roughly 20 lines deleted, including the now-unused `Vector3` import.

### 5. `test_strategies.py` — barrier stub test asserted the wrong contract

`test_barrier_is_stub` expected `BarrierStrategy.next_waypoints()` to return `[]`, but the
implementation raises `NotImplementedError`. The raise is the correct behaviour (project rule: no
silent failures — an empty plan would look like a valid "no waypoints" answer to a caller), so the
test was updated to assert the raise.

### 6. `telemetry_node.py` — cosmetic + doc fixes

- The operator summary printed `batt=-100%` when the bridge's `-1.0` "no battery message yet"
  sentinel was active. Now prints `batt=?` until a real reading arrives.
- Docstring said the node subscribes to `/detections`; the actual topic is `detection`.

### 7. Unused imports

`Vector3` (guidance_node, part of the dead-code removal) and `os` (03_train.py) removed. The
`import torch` in `03_train.py` stays — it is load-order-sensitive (Windows fbgemm.dll bug),
and commented as such.

## Flagged, deliberately not changed

### Force-arm bypasses preflight checks — `autopilot_bridge.py:_send_arm`

`param2=21196.0` with `from_external=False` replicates `commander arm -f`, skipping PX4's
preflight checks. This is intentional and commented (needed for SITL bring-up), but it is the
wrong default for a real aircraft. **Recommendation:** gate it behind a `sitl_force_arm`
parameter defaulting to `false` before any hardware flight. Decision deferred to the owner.

### Hailo output parser is a placeholder — `detector_node.py:_hailo_forward`

The YOLO output parsing (`stride = 6`, flat `[x1,y1,x2,y2,conf,cls]` assumption) is explicitly a
placeholder and must be adapted to the real `.hef` output layer shape before hardware use. The
normalised-bbox guard added earlier (`_fill_geolocation` rejects out-of-range centres loudly)
contains the blast radius if raw pixel coordinates ever land there.
